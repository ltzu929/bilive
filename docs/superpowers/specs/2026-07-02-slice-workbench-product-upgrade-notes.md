# 切片工作台产品升级确认记录

## 背景

本轮升级不是重构项目结构，而是在现有 `/tasks` 切片工作台基础上提高产品可用性。现有界面已经包含 UP 主队列、源录播预览、弹幕密度图、人工判断面板、运行任务详情和投稿流水线；后续改动应沿用这些模块，不另起一套页面。

## 已确认目标

第一阶段目标覆盖三件事：

- 审核效率：切完后能更快判断哪些片段值得继续处理。
- 启动可靠：运行中能清楚知道任务有没有跑、卡在哪里、Windows 重任务节点是否正常。
- 成片修正：选中片段后能方便调整边界、保存区间并重新渲染。

暂不把“高置信片段自动发布”作为本轮主线。自动发布涉及副作用和信任边界，先保持现有 fail-closed 逻辑。

## 主场景

主工作流定为“运行中盯进度 + 单条精品精修”：

- 第一屏优先解释运行状态、失败和可恢复动作。
- 选中失败项或候选片段后，进入对应录播和片段上下文做精修。
- 批量复盘仍保留，但不是本轮第一优先级。

## 失败恢复设计

失败恢复是本轮第一条产品主线。

- 第一屏增加“失败收件箱”，不是只在 UP 队列中弱提示。
- 失败项按阻塞程度排序：整场任务/worker 阻塞优先，其次是渲染或入队失败，再到 MiMo 判断失败，最后是投稿失败。
- 每个失败项突出一个推荐动作，但必须由用户手动确认后执行。
- 点击失败项应定位到对应 UP 主、录播和片段；视频跳到失败区间，右侧显示失败原因和恢复按钮。

现有能力需要被聚合，而不是重新发明：任务级已有排队/重跑/取消/标记完成，片段级已有重新分析/重新渲染/保存区间/保留/丢弃，worker 级已有唤醒/停止并恢复 processing。

## 精修设计

精修第一优先级是边界修正。

- 在弹幕密度图上拖拽入点和出点。
- 数字输入仍保留，作为精确兜底。
- 拖拽时只更新预览和输入框，标记为“未保存修改”。
- 保存和重渲染必须由用户手动触发，避免高频拖动造成误保存或重任务浪费。
- 拖动入点或出点时，视频应跳到对应边界并暂停，帮助判断开头和结尾是否自然。

## 当前实现原则

- 修改必须建立在现有 UI 和 API 之上。
- 保留现有 DOM ID、状态轮询、任务动作、片段动作和上传队列读法，避免破坏现有测试和运行边界。
- 优先把现有状态翻译成更明确的产品入口和推荐动作。
- 不在 Pi 上增加重任务；切片、MiMo、ASR、字幕、渲染和上传仍由 Windows 端处理。
