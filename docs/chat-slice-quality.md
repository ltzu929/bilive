# 聊天切片质量方向

## 目标

自动切片的目标从“截取弹幕高峰”调整为“挑选聊天精品片段”。

弹幕高峰只作为召回线索，表示附近可能有观众反应；真正的保留标准是片段能否独立成片、能否让陌生观众看懂并愿意看完。系统不追求稳定产量，允许一场直播没有自动投稿，也允许一场直播产出多条，只要每条都达到质量标准。

## 核心原则

一条值得发的聊天切片应该满足：

- 有明确主题，能被一句具体标题概括。
- 陌生观众不看原直播上下文也能理解。
- 有清楚的开头、发展和落点。
- 至少具备一种观看价值：完整梗或故事、明确观点、强情绪反应、弹幕互动。
- 边界服务于理解，而不是机械贴近弹幕峰值。

自动切片不是从直播里截热闹，而是从直播里挑出一个陌生观众能独立看懂、愿意看完、看完有收获或情绪反馈的聊天短片。

## 候选范围

候选范围应围绕弹幕峰值取较大的上下文。默认方向是峰值前后各约 120 秒，合并后单个候选范围最多约 5 分钟。

这个大范围只用于提供上下文，不代表最终切片要保留全部内容。MiMo 需要在大范围里寻找真正可投稿的连续短片，必要时可以返回 0 个、1 个或多个片段。

## MiMo 角色

MiMo 的角色是“短视频剪辑师 + 严格主编”。

它不只是判断候选是否精彩，而是要在较大的聊天上下文中主动寻找可投稿片段：

- 像剪辑师一样找出合适的开始和结束。
- 像主编一样判断是否值得发。
- 像普通观众一样评估理解成本。
- 像投稿运营一样生成标题、简介和标签。

故事、观点、情绪反应、弹幕互动没有固定优先级。MiMo 应选择最能独立成片、最容易被陌生观众理解、最有观看价值的一段或多段。

## 保留标准

值得保留的片段通常属于以下类型之一：

- 主播讲了一个完整梗、故事、吐槽、回忆或经历。
- 主播对某个话题给出有态度、有信息量的观点。
- 主播出现有来源的强情绪反应，例如爆笑、惊讶、破防、害羞或认真起来。
- 弹幕提问、SC、观众整活或接梗成为片段主线。

这些类型平权，最终以成片质量为准。一个强互动片段不低于一个完整观点片段；一个强情绪片段也不低于一个故事片段。关键是它是否能独立观看、是否有明确看点、是否有自然落点。

## 边界标准

切片边界服务于理解。

开头需要提供必要铺垫，让观众知道当前话题是什么；结尾需要有落点，例如笑完、答完、吐槽完、观点说完或互动结束。不要只围绕最热闹的一秒硬切。

如果一个情绪反应依赖前面的铺垫，就不能把反应单独切成一条。如果两个片段合起来才完整，应合成一个片段，而不是拆开。

## 多片段策略

一个大候选范围内可以产出多个切片，只要每个切片都独立达标。

多片段需要遵守去重原则：

- 同一话题、同一落点，只保留更完整、更好看的一条。
- 同一话题但落点不同，可以拆成多条。
- 共享铺垫但无法独立理解的片段，不应拆开。
- 轻微重叠不是问题，重复表达同一个看点才是问题。

不设置每场直播的硬数量上限。质量达标就保留，质量不达标就不自动投稿。

## 投稿信息

MiMo 应同时生成投稿标题、简介和标签，因为它最了解片段的内容核心。

标题风格采用 B 站口语标题，轻微整活但不夸张。标题必须来自片段真实内容点，不能泛泛写“直播精彩片段”，也不能制造片段里不存在的冲突。

标题方向示例：

- 弹幕一句话把主播整不会了
- 聊着聊着突然认真起来了
- 这个问题问完主播沉默了两秒
- 主播：你们这个理解角度也太怪了
- 一条 SC 引出了一段真心话

简介应比标题更稳，概括片段发生了什么，并保留直播来源说明。标签围绕主播、直播、聊天、互动类型、话题关键词和情绪类型生成。

## 严模式

严模式不是限制数量，而是限制质量。

自动投稿只发布像剪辑师主动挑出来的片段。没有达标片段时可以 0 条；同一场直播中多个片段都独立达标时也可以多条。对“有点意思但不够完整、不够好懂、不够有落点”的片段，应留给人工复核，而不是自动投稿。

## 实施基线

生产实现应保持 Pi/Windows 边界：Pi 只负责录制、仪表盘和触发 Windows；聊天切片判断、ASR、字幕、元数据和上传入队仍只在 Windows 执行。
